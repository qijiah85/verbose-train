# 文档风格指南

统一本仓库 Markdown 文档的结构、语气与排版，保证可读性与一致性。

## 基本原则

1. **读者优先**：先写结论与使用时机，再写细节
2. **可执行**：步骤、命令、检查项必须具体，避免空泛口号
3. **单一主题**：一篇文档只覆盖一个主题；过大则拆分
4. **中英混排规范**：正文默认简体中文；命令、路径、API、专有名词保留英文原文

## 文档骨架

### 规范文档

建议按以下顺序组织：

1. 标题（`#`）
2. YAML frontmatter（见命名规范）
3. 摘要（2–5 句：目的、适用范围、非目标）
4. 何时使用 / 何时不使用
5. 规范正文（分节）
6. 示例
7. 检查清单（可选）
8. 相关文档
9. 修订记录

### Agent 技能（`SKILL.md`）

建议按以下顺序组织：

1. YAML frontmatter（`name`、`description`）
2. 概述（技能解决什么问题）
3. 适用场景
4. 步骤流程（编号列表，可含决策点）
5. 红线 / 禁止事项
6. 示例或对照表
7. 辅助文件链接（如有）

## Markdown 约定

| 元素 | 约定 |
|------|------|
| 标题 | 使用 ATX 风格（`#` / `##`）；每文档仅一个 `#` |
| 标题层级 | 不要跳级（如 `##` 后直接 `####`） |
| 列表 | 同级列表符号一致；有序步骤用数字列表 |
| 强调 | 少用加粗；仅强调关键术语或警告 |
| 代码 | 行内用反引号；代码块标注语言（`bash`、`yaml`、`markdown`） |
| 链接 | 优先相对路径；链接文本说明目标，避免「点击这里」 |
| 表格 | 用于对照、枚举、字段说明；避免超宽表 |
| 图片 | 如需插图，放在同目录 `assets/`，文件名 `kebab-case` |

## 语气与用词

- 使用直接、祈使语气写步骤：「先运行测试，再修改实现」
- 避免模糊词：「适当」「尽量」「视情况」——改为可判定条件
- 状态词统一：`draft`（草稿）、`active`（现行）、`deprecated`（废弃）
- 版本号使用语义化版本：`MAJOR.MINOR.PATCH`

## 中文排版建议

- 中英文混排：英文单词两侧加空格，如「使用 `git` 提交」
- 全角标点用于中文句子；代码与英文短语内用半角
- 不要用 emoji 作为结构标记

## 质量自检清单

提交前确认：

- [ ] 标题与文件名语义一致
- [ ] frontmatter / 必填字段完整
- [ ] 有明确的适用范围与非目标（规范）或触发场景（技能）
- [ ] 步骤可照做，无缺失前置条件
- [ ] 相对链接可访问
- [ ] 无 `TBD` / `TODO` 占位残留（除非状态为 `draft` 且已标明）
- [ ] 与现有文档无未说明的冲突
