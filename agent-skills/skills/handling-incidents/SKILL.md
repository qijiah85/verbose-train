# Handling Incidents

## 概述

线上故障时以**减轻影响**为第一目标：检测 → 缓解 → 恢复 → 复盘。未缓解前不做大规模重构或无关「优化」。

## 适用场景

**使用，当：**

- 生产错误率/延迟飙升、核心链路不可用
- 需要回滚、降级、限流或开关止血
- 故障后需要写复盘与改进项

**不要使用，当：**

- 本地开发环境问题或未合并代码的 bug（用 `systematic-debugging`）
- 无用户影响的清理/重构任务

## 流程

1. **声明事故**：严重级别、影响范围（谁、什么功能、开始时间）、当前指挥角色。
2. **止血优先**：选最快可逆动作——回滚、关开关、降级、限流、failover；并行保留证据（日志、指标、变更窗口）。
3. **沟通**：向干系人同步现状、下一步、下次更新时间；避免沉默。
4. **定位**：在影响已可控后，用时间线关联「变更 × 症状」；需要时再用系统化调试。
5. **恢复验证**：用真实监测与关键用户路径确认恢复，而非「感觉好了」。
6. **复盘（blameless）**：
   - 时间线
   - 影响
   - 触发因素与放大因素
   - 已做缓解
   - 改进项（检测、预防、响应），每项有负责人与期限
7. **若暴露需求/设计缺口**：复盘后转入 `gathering-requirements` 或设计技能，避免只修表象。

### 决策点

| 条件 | 行动 |
|------|------|
| 缓解与深挖冲突 | 先缓解；深挖在单独线程/人员 |
| 不确定能否回滚 | 评估数据兼容；优先功能开关关闭 |
| 误报 / 无影响 | 降级为普通 bug 流程并记录 |

## 红线

- 禁止在事故高峰做无关重构或依赖升级
- 禁止无证据声称「已恢复」
- 禁止复盘归咎个人而忽略系统因素
- 禁止删除故障窗口内的日志与变更记录

## 交付物检查

- [ ] 影响与缓解措施已记录
- [ ] 恢复有监测证据
- [ ] 复盘改进项可执行且有主责
- [ ] 需跟进的需求/设计债务已登记

## 示例

### 输入

「生产上收藏接口全 500，刚才刚发版。」

### 期望行为

立即评估回滚或关开关，同步影响，保留发版与错误样本，恢复后再写时间线复盘，而不是先在生产热修重构收藏逻辑。
