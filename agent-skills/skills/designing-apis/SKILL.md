# Designing APIs

## 概述

把能力沉淀为**清晰、可演进的接口契约**：资源/操作、请求响应、错误模型、幂等与版本策略。契约先于（或同步于）实现，避免前后端各自猜测。

## 适用场景

**使用，当：**

- 新增或变更对外/对内 API
- 需要统一错误码、分页、鉴权头等横切约定
- 讨论破坏性变更与兼容策略

**不要使用，当：**

- 系统边界与模块职责尚未定（先 `designing-architecture`）
- 纯内部函数重构且无跨进程契约

## 流程

1. **列出用例**：每个 API 对应哪些验收标准（追溯到需求）。
2. **选风格并保持一致**：资源型 REST、RPC、事件等；同一边界内不混用两套语义。
3. **定义契约字段**：
   - 路径/方法或过程名
   - 请求/响应 schema（类型、必填、示例）
   - 鉴权与权限
   - 幂等键 / 重试语义（写操作必答）
4. **错误模型**：稳定错误码、HTTP 映射、是否可重试、用户可理解信息与内部细节分离。
5. **版本与兼容**：何种变更算破坏性；如何废弃（deprecation）窗口。
6. **写出最小示例**（请求/响应各一）并对照验收标准。
7. **进入实现**时用 Superpowers TDD：先契约测试或消费者期望，再写处理器。

### 决策点

| 条件 | 行动 |
|------|------|
| 已有项目 API 规范 | 遵循 `specifications/api/` 中现行文档 |
| 仅内部服务间调用 | 仍要契约，但可简化公开文档 |
| 破坏性变更 | 新版本并行或显式迁移计划，禁止静默改语义 |

## 红线

- 禁止同一含义多种错误码、同一错误码多种含义
- 禁止在无版本策略时随意重命名/删字段
- 禁止把内部堆栈或敏感数据返回给客户端

## SOP 输入 / 输出

- **输入：** `A-PRD`；建议同时有 `A-ARCH` / `A-DATA`
- **输出：** `A-API`（模板 `agent-skills/templates/artifacts/a-api.md`）

## 阶段门禁

缺验收标准时不得「先写一堆 URL」；缺数据模型时标出假设。

## 交付物检查

- [ ] 每个写操作有幂等/重试说明
- [ ] 错误模型完整
- [ ] 至少一组成功与失败示例
- [ ] 与需求验收标准可逐条对应
- [ ] `A-API` 已落盘

## 示例

### 输入

「收藏商品的接口怎么定？」

### 期望行为

给出如 `PUT /v1/users/me/favorites/{productId}` 等契约、401/404/409 语义、幂等说明，以及列表查询的分页约定，而不是先写控制器代码。
